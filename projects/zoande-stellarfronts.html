StellarFronts

A browser-based real-time space strategy game. Login, claim a country, explore the galaxy, expand with starbases and districts, run an economy and research over time. React frontend, BabylonJS 7 command view, WebSocket game server, separate auth server, and an orchestrator that hosts multiple game versions in parallel.

StellarFronts is built around logistics, expansion, economy, research, market trading, and system control - not direct combat. The local stack runs end-to-end with one command (npm run dev:all) and brings up the React client, the HTTP auth server, and the WebSocket game server together.

Architecture

The codebase is split clean down the middle: src/ is the React UI, gameplay shared types, and BabylonJS scene; server/ is the auth server, the game server, and the orchestrator that ties them together behind a single gateway.

The auth server handles login, signup, sessions, and the game catalog. It uses HttpOnly cookies, PBKDF2-hashed passwords, and a SQLite store on disk. Google and Microsoft OAuth endpoints are stubbed for later.

The game server is a WebSocket process that owns the live simulation, broadcasts state, and saves dirty state on a timer to JSON per game.

The orchestrator is the interesting part - it checks out each game version as an isolated git worktree and runs it as its own game-server process, so a brand-new game can run updated code while an older live game stays pinned to the code it was started on. Clients still talk to one public endpoint; the orchestrator routes them to the right version.

What you can do in-game

Claim a country

Normal accounts pick from generated countries on join. Observers and admins can enter games without claiming.

Explore the galaxy

BabylonJS 7 command view with a procedural skybox and loaded GLB ship and starbase models, mixed with HUD-style overlays.

Expand

Build starbases and districts to extend reach and capacity over the long game.

Run an economy

Logistics, research, and market trading are first-class systems - the game is balanced around them rather than around combat.

Hold sessions

Game state persists per-game on a dirty-write timer. You can leave and resume a country across sessions.

Admin debug

Admin accounts get in-game debug commands, full map visibility, and a developer panel at /dev.

Stack

React 19 + Vite

Pathname-driven flows in App.tsx: login at /, catalog at /home, command view at /game/:gameId, dev panel at /dev.

BabylonJS 7

The 3D command view, procedural space skybox, and loaded GLB ships/starbases. Asset warm-up runs on the auth screen so the scene boots fast.

Node + WebSocket

The game server is a long-lived Node process. Auth is a separate HTTP server so login can't take the game server down.

SQLite + JSON state

Accounts and dev stats live in SQLite. Per-game state writes to server/state/games/<id>/game-state.json on a dirty-write timer.

Orchestrator

One gateway, many game-server versions running side by side on isolated git worktrees. Existing games stay on the code they started on.

Vercel Analytics

Lightweight client-side analytics on the React app for usage signals.

Status

Actively in development. The full local stack runs end-to-end and the core loop - login, claim, explore, expand, run an economy, persist - is wired up. Systems and balance are explicitly still a work in progress. OAuth endpoints exist but return 501 until Google and Microsoft are turned on.
